First-Time Setup

Two prompts

Getting a fresh Replit project ready for Codex is just two steps: one shell command for git, and one prompt you hand to Codex that builds all your context files at once. After that, your daily workflow is just the Kick-Off / Checkpoint / Wrap-Up loop.

ℹ️
Only needed once per project. Codex auth persists in ~/.codex/ across sessions, and your AGENTS.md / CONTEXT.md / ROADMAP.md live in the repo, so once they're in place you're set.
★ Step 1 — Shell

Connect the project to GitHub

First create an empty repo at github.com/new (no README, no gitignore, no license). Then paste this one block into the Replit Shell, edit the three values at the top, and run it. It configures git, links the remote, and pushes your first commit in one go.

bash (run in Replit Shell)
# --- edit these three lines ---
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
REPO_URL="https://github.com/your-user/your-repo.git"
# -------------------------------

git init
git branch -M main
git remote add origin "$REPO_URL"
git add .
git commit -m "initial commit"
git push -u origin main
💡
When prompted for a password on push, use a Personal Access Token, not your GitHub password (disabled since 2021). Make one at github.com/settings/tokens with repo scope. Store it as a Replit Secret named GITHUB_TOKEN to avoid re-entering it.
★ Step 2 — Codex

Let Codex build your context files

Launch codex in your project root, then paste this single prompt. Codex inspects your project and writes all three context files plus a .gitignore, then commits them. One prompt does what used to be eight steps.

paste into Codex

   read from environment variables via Replit Secrets), always
   run tests after changing core logic, and reference
   CONTEXT.md and ROADMAP.md.

3. Create CONTEXT.md with sections: Current Status, Current

Show me each file before committing so I can confirm.

Safety & .gitignore

Read this

Codex in workspace-write or danger-full-access can touch any file in your project, and it commits whatever you tell it to. A good .gitignore is your seatbelt: it stops secrets, build junk, and session files from ever reaching GitHub.

⚠️
The single most common mistake is committing a .env file with live API keys. Once it's pushed to a public repo, treat those keys as compromised even if you delete the file later, git history keeps them. Set up .gitignore before your first commit, not after.

Starter .gitignore for a Node / React / Replit project

.gitignore
# Secrets and environment
.env
.env.*
*.pem
*.key
secrets.json

# Dependencies
node_modules/
.pnpm-store/

# Build output
dist/
build/
.next/
.vite/

# Codex / agent session files
.codex/
*.session.jsonl

# Replit
.replit
replit.nix
.config/
.upm/
.cache/

# Logs and OS cruft

Thumbs.db

# Editor

Where secrets should actually live

Never hardcode keys in your code or AGENTS.md. Use Replit Secrets (the lock icon in the sidebar), which inject as environment variables at runtime. Reference them in code as process.env.YOUR_KEY, never the literal value.

💡
Add a rule to your AGENTS.md so Codex respects this automatically: Never hardcode secrets or API keys. Always read them from environment variables via Replit Secrets. Codex will then refuse to inline a key even if you ask it to in passing.

If you already committed a secret

Rotate the key immediately at the provider (regenerate it so the old one stops working). Removing it from git history is secondary, the rotation is what actually protects you. Then add the file to .gitignore and commit that.

🔒
Codex hardens itself too. Recent releases tightened command safety: /diff no longer runs repository-provided Git helpers or hooks, PowerShell parser execution is blocked on non-Windows hosts, and browser-origin exec-server handshakes are rejected. These close paths where a malicious repo could trick the agent into running code. Keep Codex updated (npm i -g @openai/codex) to stay current on these.

Connecting MCP Servers

v0.134 updated

MCP (Model Context Protocol) servers give Codex extra tools: database access, web search, third-party APIs, your own custom tooling. v0.134 added per-server environment targeting and OAuth support for streamable HTTP servers, so the setup is cleaner than it used to be.

Add a server to config.toml

MCP servers are declared in ~/.codex/config.toml under [mcp_servers.NAME] blocks. Each one can be a local command (stdio) or a remote HTTP endpoint.

~/.codex/config.toml
# Local stdio server (runs as a subprocess)
[mcp_servers.filesystem]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-filesystem", "/home/runner/workspace"]

# Remote streamable HTTP server with OAuth (v0.134+)
[mcp_servers.my_api]
url = "https://mcp.example.com/sse"
bearer_token_env_var = "MY_API_TOKEN"

# Per-server environment targeting (v0.134+)
[mcp_servers.database]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-postgres"]
env = { DATABASE_URL = "${DATABASE_URL}" }
ℹ️
Use bearer_token_env_var and env rather than pasting tokens directly into config.toml. Store the actual values as Replit Secrets so they never end up in a file that could be committed.

Verify your MCP servers loaded

After editing config.toml, codex doctor reports which MCP servers connected and which failed, with error details. Run it before assuming a tool is unavailable.

bash
codex doctor
💡
In Replit, stdio servers that need network access (like npx fetching a package) can be blocked by Codex's sandbox. If a server fails to start, try launching Codex with --sandbox danger-full-access for that session, the Replit container is itself the outer sandbox so it stays reasonably safe.
ℹ️
As of v0.142, Codex uses tool search by default for MCP tools where the model supports it. Instead of loading every tool's full schema upfront, Codex discovers them on demand, which keeps your context budget lean when you have several MCP servers wired up. Older models and providers fall back to the previous behaviour automatically.

AGENTS.md

Project Context

AGENTS.md is the project context file that Codex reads before every task. It tells Codex about your stack, your conventions, and your rules. Keep it accurate and it'll save you re-explaining the project session after session.

ℹ️
Files closer to your working directory take priority. The load order is: ~/.codex/AGENTS.override.md (global override), then ~/.codex/AGENTS.md (global default), then git root up to current directory: AGENTS.override.md, then AGENTS.md. Max size is 32 KiB by default.

Three-File Setup (Recommended)

Split your project context across three files to keep things manageable. AGENTS.md holds the stable stuff, CONTEXT.md tracks the moving stuff, ROADMAP.md tracks the plan.

★ Recommended Structure

AGENTS.md (project root)

Architecture, conventions, commands to always run. The stuff Codex needs every session.

CONTEXT.md (project root)

Current sprint focus, recent decisions, what's in progress. Update this each day.

ROADMAP.md (project root)

Feature list and priorities. Reference Codex to it when planning work.

markdown
## Project: YourProjectName

### Stack
- Frontend: React 18 + TypeScript
- Backend: Node.js / Express
- Database: PostgreSQL (via Prisma or pg)
- Hosting: Replit (dev), Vercel + Railway (prod)

### Conventions
- Use camelCase for JS/TS variables
- Async/await over .then() chains
- Component files: PascalCase.tsx
- API route files: kebab-case.js

### Commands
- Start dev server: npm run dev
- Run tests: npm test
- Type check: npx tsc --noEmit

### Rules
- Never delete files without asking first
- Always run npm test after modifying core logic
- Keep .env variables in Replit Secrets, not hardcoded
- Commit messages: conventional commits format

### Context files
- See CONTEXT.md for current sprint
- See ROADMAP.md for feature priorities

Archive Old Sessions

As sessions pile up, your resume list gets noisy. Archiving tucks a session away so it stays out of codex resume until you restore it. Archived sessions are protected from accidental resume or fork. Use /archive inside the TUI, or the shell commands below.

bash
# Archive a session (keeps it but hides it from resume)
codex archive

# Bring one back
codex unarchive
💡
Archiving only hides a session. If you want a session gone for good (v0.140+ added permanent deletion), that removes it from your local history entirely and can't be undone, so reach for archive unless you specifically need it erased.

Approval Modes

Codex has three sandbox modes that control what it can do without asking. In Replit you're already inside a container, so workspace-write is the sweet spot for daily work.

🔵
Read Only
--sandbox read-only
Codex can read files and suggest changes but cannot write anything. You approve every file edit individually. Slowest but safest for sensitive codebases.
🟡
Workspace Write
--sandbox workspace-write
Codex can read and write files in your project, but shell commands still need approval. Best for daily Replit work. Recommended default.
🔴
Full Access
--sandbox danger-full-access
Codex can read, write, and run shell commands without approval. Use for trusted automation tasks. Not recommended for exploratory sessions.

Approval Policy (separate from sandbox)

The approval policy controls when Codex asks for human confirmation regardless of sandbox mode.

~/.codex/config.toml
model = "gpt-5.5"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
web_search = "cached"

Create named profiles in your config.toml and switch between them with --profile name. Useful if you work on multiple projects with different trust levels.

config.toml
[profiles.daily]
model = "gpt-5.5"
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[profiles.safe]
approval_policy = "untrusted"
sandbox_mode = "read-only"

[profiles.auto]
approval_policy = "never"
sandbox_mode = "danger-full-access"
bash
# Load a profile:
codex --profile daily

Skills

Reusable workflows

A skill is a folder with a SKILL.md file (name and description in the frontmatter) plus optional scripts. Drop it in ~/.codex/skills/, restart Codex, and it loads automatically. Codex matches your prompt against installed skill descriptions and pulls the right one in. Write the instructions once, every future session inherits the behaviour.

ℹ️
Skills use the open SKILL.md standard, so the same skill works across Codex, Claude Code, OpenClaw, and Cursor. The only difference is the folder each agent reads from. That cross-compatibility is the whole point of the standard.

Where skills live

ScopePath
Global (all projects)~/.codex/skills/
Project-only (in the repo).codex/skills/
💡
Codex won't re-scan the skills folder mid-session. After adding or editing a skill, restart Codex (or start a new session) for it to load.

One skills folder across all your agents

Since the format is identical, you can symlink a single skills folder so Codex, Claude Code, and OpenClaw all share one set. Maintain skills in one place, every agent inherits them.

bash
# Point Codex and OpenClaw at your Claude skills folder
ln -s ~/.claude/skills/ ~/.codex/skills
ln -s ~/.claude/skills/ ~/.openclaw/skills

Writing a Good AGENTS.md

  • Keep it factual, not aspirational. Document what the project IS, not what you hope it will be. Codex uses this as ground truth.
  • Include commands it can run. List the actual test, lint, and build commands for your project. Codex will use them without being asked.
  • Write explicit rules. "Never delete files without asking" is a valid rule. Put it in.
  • Name your files and folders. If Codex knows your folder structure upfront it navigates faster and makes fewer wrong-directory errors.
  • Stay under 32 KiB. The default max is 32 KiB. If your project is complex, split context across AGENTS.md + CONTEXT.md + ROADMAP.md rather than cramming it all in one file.
  • Update it when things change. AGENTS.md is only useful if it's current. When you change your stack or rename a folder, update the file.
  • Use override files sparingly. AGENTS.override.md at the global level can break project-specific context. Use it only for truly global personal preferences.
⚠️

Common Mistakes

1. Using browser OAuth flow in Replit shell

Running plain codex auth login tries to open a browser, which doesn't work over SSH. Always use --device flag in Replit shell: codex auth login --device

2. Starting fresh instead of resuming

After a Replit reload, running codex starts a brand new session with no history. Use codex resume --last to pick up where you left off. You won't lose context.

3. No AGENTS.md in the project root

Without an AGENTS.md, Codex has zero project context and you'll spend the first 10 minutes of every session re-explaining your stack. Run /init once and clean it up manually.

4. Full-access mode left on by default

Leaving danger-full-access as your permanent default means Codex can run arbitrary shell commands without asking. Fine for specific automation tasks, risky as a daily driver. Use workspace-write as your default.

5. AGENTS.md over 32 KiB

If your AGENTS.md hits the 32 KiB limit, Codex silently truncates it. Split content across CONTEXT.md and ROADMAP.md. Or bump the limit in config: project_doc_max_bytes = 65536

Troubleshooting

🩺
First port of call for any issue: run codex doctor in your Replit shell. It checks your installation, auth status, sandbox config, MCP servers, and network access in one pass. Often tells you exactly what's broken before you have to guess.

If Codex keeps asking you to log in, the auth token is probably corrupted or expired. Run codex doctor first to confirm, then clear and re-auth:

bash
codex auth logout
codex auth login --device

If Codex keeps asking "approve this?" for every file write, you're probably still on the default read-only sandbox. Switch to workspace-write either in config or on launch:

bash
codex --sandbox workspace-write

Codex blocks network by default inside its sandbox (OS-level Landlock on Linux). This is expected behaviour. If a task needs network access (e.g. running npm install), enable it in config or pass the flag:

bash
codex --sandbox danger-full-access
# or for specific network use, run npm install manually first
💡
In Replit specifically, you're already inside a container with controlled network access. The double-sandboxing can cause unexpected blocks. Running danger-full-access in Replit is relatively safe since the container itself is the outer sandbox.

Codex loads AGENTS.md from the git root and current directory. If it's not picking up your file, check:

  • You're launching Codex from the correct directory (project root, not a subdirectory)
  • The file is named exactly AGENTS.md (case-sensitive)
  • The file is under 32 KiB (check with wc -c AGENTS.md)
  • You don't have a conflicting ~/.codex/AGENTS.override.md that's overriding it

## Project: MyApp

### Stack
- Frontend: React 18 + TypeScript, Tailwind CSS
- Backend: Node.js 20 + Express
- Database: PostgreSQL (via node-postgres / pg)
- Auth: JWT tokens stored in httpOnly cookies
- Hosting: Replit (dev), Railway (prod API), Vercel (prod frontend)

### Folder structure
- /src/components - React components
- /src/pages - Page-level components
- /src/api - Express route handlers
- /src/db - Database queries and schema
- /src/lib - Shared utilities

### Commands
- Start dev: npm run dev
- Run tests: npm test
- Type check: npx tsc --noEmit
- Lint: npm run lint

### Coding conventions
- Async/await, never .then() chains
- camelCase for variables and functions
- PascalCase for components and types
- snake_case for database column names
- No inline styles, use Tailwind classes only

### Hard rules
- Never delete files without asking first
- Never hardcode secrets or API keys
- Always run tests before marking a task done
- Keep Replit Secrets as the source for env vars
- Commit messages use conventional commits format

### Context files
- CONTEXT.md: current sprint focus and what is in progress
- ROADMAP.md: feature backlog and priorities
