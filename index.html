Install & Auth

★ Daily

One npm command installs Codex globally in your Replit shell. Authentication uses your existing ChatGPT subscription, zero extra cost.

💡
Codex CLI is included in all ChatGPT plans (Plus $20/mo, Pro $100-$200/mo). Using OAuth means you pay nothing extra per token. API key mode is also supported but costs more for typical daily use.
★ Start Here

Fresh Install (Replit Shell)

bash
npm i -g @openai/codex

Requires Node.js 18+. Replit environments come with Node already. Codex CLI is built in Rust so it's fast, but installs via npm.

Authentication

★ Recommended for Replit

Device Code Flow (SSH-safe)

Replit shell runs over SSH, so a browser doesn't pop open. Use device-code auth instead: Codex prints a short code, you visit a URL on your phone or laptop, done.

bash
codex auth login --device

Visit the printed URL, enter the code, authorize with your ChatGPT account. Auth token is saved to ~/.codex/ and persists across Replit sessions.

Alternative: API Key

OPENAI_API_KEY

If you have a platform.openai.com API key, set it as a Replit secret. Better for CI pipelines, but costs more per token than a ChatGPT subscription for normal use.

bash
# Add to Replit Secrets as OPENAI_API_KEY
# then launch with:
codex

Verify Install

bash
codex --version
codex auth status

AGENTS.md

Project Context

AGENTS.md is the project context file that Codex reads before every task. It tells Codex about your stack, your conventions, and your rules. Keep it accurate and it'll save you re-explaining the project session after session.

ℹ️
Files closer to your working directory take priority. The load order is: ~/.codex/AGENTS.override.md (global override), then ~/.codex/AGENTS.md (global default), then git root up to current directory: AGENTS.override.md, then AGENTS.md. Max size is 32 KiB by default.

Three-File Setup (Recommended)

Split your project context across three files to keep things manageable. AGENTS.md holds the stable stuff, CONTEXT.md tracks the moving stuff, ROADMAP.md tracks the plan.

★ Recommended Structure

AGENTS.md (project root)

Architecture, conventions, commands to always run. The stuff Codex needs every session.

CONTEXT.md (project root)

Current sprint focus, recent decisions, what's in progress. Update this each day.

ROADMAP.md (project root)

Feature list and priorities. Reference Codex to it when planning work.

markdown
## Project: YourProjectName

### Stack
- Frontend: React 18 + TypeScript
- Backend: Node.js / Express
- Database: PostgreSQL (via Prisma or pg)
- Hosting: Replit (dev), Vercel + Railway (prod)

### Conventions
- Use camelCase for JS/TS variables
- Async/await over .then() chains
- Component files: PascalCase.tsx
- API route files: kebab-case.js

### Commands
- Start dev server: npm run dev
- Run tests: npm test
- Type check: npx tsc --noEmit

### Rules
- Never delete files without asking first
- Always run npm test after modifying core logic
- Keep .env variables in Replit Secrets, not hardcoded
- Commit messages: conventional commits format

### Context files
- See CONTEXT.md for current sprint
- See ROADMAP.md for feature priorities

Auto-generate with /init

Start a Codex session and run /init to have Codex scan your project and scaffold an AGENTS.md automatically. Good starting point, then edit from there.

bash
# Start a session in your project directory
codex

# Then inside the session:
/init

Morning Kick-Off Prompt

★ Daily

Run this at the start of every session to orient Codex without re-explaining the whole project. It reads your AGENTS.md and CONTEXT.md, then asks what you want to work on.

bash (run in shell)
codex "Read AGENTS.md and CONTEXT.md first. Then confirm:
1. The current project context you have loaded
2. What was last worked on
3. Any open issues or blockers you can see

Do not start writing code yet. Wait for me to tell you what to work on today."
💡
If you're resuming a previous session (rather than starting fresh), use codex resume --last instead. Codex will reload the last session's context automatically. See the Checkpoint section below.
⚡

Checkpoint / Session Resume

★ Daily

Codex sessions end when you close the shell, but the conversation history is stored locally and you can pick it back up cleanly. Use resume after a Replit reload or whenever you want to continue where you left off.

★ Recommended

Resume Last Session

bash
codex resume --last

Picks up your most recent session. Codex reloads the conversation history and continues where you left off. Best option after a Replit reload or shell disconnect.

Pick From History

bash
codex resume

Shows an interactive picker with your recent sessions. Useful if you've been running multiple projects and want to choose the right one.

Mid-Session Compact (Context Management)

GPT-5.5 gives you a 400K context window in Codex, which is generous but not unlimited. If a session gets long, use /compact inside the session to compress history and free up context budget.

slash command (inside session)
/compact

Manual Checkpoint Prompt

Codex doesn't have a dedicated checkpoint command. Instead, ask Codex to summarise progress directly into your CONTEXT.md so the next session has a clean handoff.

prompt (inside session)
Pause. Update CONTEXT.md with:
1. What we completed this session
2. What is still in progress
3. Any blockers or decisions made
4. What to pick up next session

Keep it concise, bullet points are fine.

End-of-Day Wrap-Up

★ Daily

Two-tier system. Quick checkpoint for short pauses, full wrap-up at end of day. Both update your context files so tomorrow's session starts cleanly.

⚡ Quick Checkpoint (mid-session pause)

prompt
Quick checkpoint: summarise what we just completed and what is still in progress. Update CONTEXT.md.

🌙 Full End-of-Day Wrap

prompt
End of day wrap-up. Please:
1. Update CONTEXT.md with today's progress, decisions, and blockers
2. Update ROADMAP.md if any priorities changed
3. List the top 3 things to pick up tomorrow
4. Flag any files that were changed today so I can review before committing

Do not start any new work. Just wrap up and summarise.
✓
After wrapping up, always review changed files before committing to GitHub. Even with approval modes active, Codex can make unexpected changes to files outside the scope of a task.
🔒

Approval Modes

Codex has three sandbox modes that control what it can do without asking. In Replit you're already inside a container, so workspace-write is the sweet spot for daily work.

🔵
Read Only
--sandbox read-only
Codex can read files and suggest changes but cannot write anything. You approve every file edit individually. Slowest but safest for sensitive codebases.
🟡
Workspace Write
--sandbox workspace-write
Codex can read and write files in your project, but shell commands still need approval. Best for daily Replit work. Recommended default.
🔴
Full Access
--sandbox danger-full-access
Codex can read, write, and run shell commands without approval. Use for trusted automation tasks. Not recommended for exploratory sessions.

Approval Policy (separate from sandbox)

The approval policy controls when Codex asks for human confirmation regardless of sandbox mode.

~/.codex/config.toml
model = "gpt-5.5"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
web_search = "cached"

Create named profiles in your config.toml and switch between them with --profile name. Useful if you work on multiple projects with different trust levels.

config.toml
[profiles.daily]
model = "gpt-5.5"
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[profiles.safe]
approval_policy = "untrusted"
sandbox_mode = "read-only"

[profiles.auto]
approval_policy = "never"
sandbox_mode = "danger-full-access"
bash
# Load a profile:
codex --profile daily

Slash Commands

Slash commands run inside an active Codex session (after launching with codex). Use them to scaffold context files, switch models, manage context budget, and set goals.

/init Interactive

Scans your project and auto-generates an AGENTS.md scaffold. Run this once when setting up a new project to save yourself writing the whole file from scratch.

/model Mid-session

Switch models without leaving the session. Useful for dropping to a faster/cheaper model for repetitive tasks, or bumping up to GPT-5.5 for complex architecture decisions.

/compact Context

Compresses the session history to free up context budget. Run this when the conversation gets long to avoid hitting the 400K token limit mid-task.

/goal Persistent

Sets a persisted work goal for the session. Codex keeps this goal in mind across the conversation, which helps it avoid going off-track on long multi-step tasks.

Alt+, Shortcut

Lower the reasoning effort level on the fly. Good when you want quick file edits without waiting for deep reasoning. Opposite: Alt+. raises effort.

codex exec Non-interactive

Run a one-shot Codex task from the shell without opening the interactive TUI. Great for quick scripted tasks or CI pipelines. Short form: codex e

bash
codex exec "run tests and fix any failures"

Writing a Good AGENTS.md

  • Keep it factual, not aspirational. Document what the project IS, not what you hope it will be. Codex uses this as ground truth.
  • Include commands it can run. List the actual test, lint, and build commands for your project. Codex will use them without being asked.
  • Write explicit rules. "Never delete files without asking" is a valid rule. Put it in.
  • Name your files and folders. If Codex knows your folder structure upfront it navigates faster and makes fewer wrong-directory errors.
  • Stay under 32 KiB. The default max is 32 KiB. If your project is complex, split context across AGENTS.md + CONTEXT.md + ROADMAP.md rather than cramming it all in one file.
  • Update it when things change. AGENTS.md is only useful if it's current. When you change your stack or rename a folder, update the file.
  • Use override files sparingly. AGENTS.override.md at the global level can break project-specific context. Use it only for truly global personal preferences.
⚠️

Common Mistakes

1. Using browser OAuth flow in Replit shell

Running plain codex auth login tries to open a browser, which doesn't work over SSH. Always use --device flag in Replit shell: codex auth login --device

2. Starting fresh instead of resuming

After a Replit reload, running codex starts a brand new session with no history. Use codex resume --last to pick up where you left off. You won't lose context.

3. No AGENTS.md in the project root

Without an AGENTS.md, Codex has zero project context and you'll spend the first 10 minutes of every session re-explaining your stack. Run /init once and clean it up manually.

4. Full-access mode left on by default

Leaving danger-full-access as your permanent default means Codex can run arbitrary shell commands without asking. Fine for specific automation tasks, risky as a daily driver. Use workspace-write as your default.

5. AGENTS.md over 32 KiB

If your AGENTS.md hits the 32 KiB limit, Codex silently truncates it. Split content across CONTEXT.md and ROADMAP.md. Or bump the limit in config: project_doc_max_bytes = 65536

Troubleshooting

If Codex keeps asking you to log in, the auth token is probably corrupted or expired. Clear it and re-auth:

bash
codex auth logout
codex auth login --device

If Codex keeps asking "approve this?" for every file write, you're probably still on the default read-only sandbox. Switch to workspace-write either in config or on launch:

bash
codex --sandbox workspace-write

Codex blocks network by default inside its sandbox (OS-level Landlock on Linux). This is expected behaviour. If a task needs network access (e.g. running npm install), enable it in config or pass the flag:

bash
codex --sandbox danger-full-access
# or for specific network use, run npm install manually first
💡
In Replit specifically, you're already inside a container with controlled network access. The double-sandboxing can cause unexpected blocks. Running danger-full-access in Replit is relatively safe since the container itself is the outer sandbox.

Codex loads AGENTS.md from the git root and current directory. If it's not picking up your file, check:

  • You're launching Codex from the correct directory (project root, not a subdirectory)
  • The file is named exactly AGENTS.md (case-sensitive)
  • The file is under 32 KiB (check with wc -c AGENTS.md)
  • You don't have a conflicting ~/.codex/AGENTS.override.md that's overriding it

Example AGENTS.md (Full)

AGENTS.md
## Project: MyApp

### Stack
- Frontend: React 18 + TypeScript, Tailwind CSS
- Backend: Node.js 20 + Express
- Database: PostgreSQL (via node-postgres / pg)
- Auth: JWT tokens stored in httpOnly cookies
- Hosting: Replit (dev), Railway (prod API), Vercel (prod frontend)

### Folder structure
- /src/components - React components
- /src/pages - Page-level components
- /src/api - Express route handlers
- /src/db - Database queries and schema
- /src/lib - Shared utilities

### Commands
- Start dev: npm run dev
- Run tests: npm test
- Type check: npx tsc --noEmit
- Lint: npm run lint

### Coding conventions
- Async/await, never .then() chains
- camelCase for variables and functions
- PascalCase for components and types
- snake_case for database column names
- No inline styles, use Tailwind classes only

### Hard rules
- Never delete files without asking first
- Never hardcode secrets or API keys
- Always run tests before marking a task done
- Keep Replit Secrets as the source for env vars
- Commit messages use conventional commits format

### Context files
- CONTEXT.md: current sprint focus and what is in progress
- ROADMAP.md: feature backlog and priorities
